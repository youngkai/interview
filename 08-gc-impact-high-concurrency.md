# GC 对高并发系统的影响

## 30 秒回答

Go 的 GC 是并发标记清扫，目标是在程序运行的同时完成大部分垃圾回收工作，减少 STW 时间。对高并发系统来说，GC 主要影响延迟、CPU 使用率和吞吐。对象分配越频繁、堆越大、指针越多，GC 扫描和回收成本越高。

优化方向不是关闭 GC，而是减少不必要分配、复用对象、控制堆增长、降低指针数量、避免内存泄漏，并用 pprof、trace、runtime metrics 观察 GC 暂停、堆大小、分配速率和 CPU 占用。

## GC 的基本目标

GC 负责自动回收不再使用的堆内存，降低手动内存管理复杂度。

但 GC 不是免费能力，它会消耗：

- CPU：标记、扫描、清扫对象。
- 内存：堆增长、GC 元数据、对象存活。
- 延迟：STW 暂停和调度影响。

## Go GC 的核心特点

Go GC 可以概括为：

- 并发标记清扫。
- 三色标记。
- 写屏障。
- 短 STW。
- 非分代 GC。

面试中不需要把 runtime 细节背得很死，但要说清楚它如何影响高并发。

## 三色标记简述

GC 标记阶段会从根对象出发，找到所有可达对象。

三种颜色：

- 白色：尚未确认存活，回收候选。
- 灰色：已发现，但其引用对象还没扫描完。
- 黑色：已扫描完成，确认存活。

标记结束后，仍然是白色的对象就可以被回收。

## 为什么需要写屏障

GC 标记和用户程序并发执行时，用户程序可能修改对象引用关系。

如果没有写屏障，GC 可能漏标对象，错误回收仍然在使用的对象。

写屏障的作用是记录并发标记期间的指针变更，保证标记正确性。

## STW 是什么

STW 是 Stop The World，即暂停所有用户 Goroutine。

Go GC 大部分工作并发执行，但仍有短暂 STW 阶段，例如：

- GC 开始时做准备。
- GC 结束时完成某些收尾。

STW 时间通常较短，但在低延迟系统中仍然需要关注。

## GC 对高并发系统的影响

### 1. 延迟抖动

GC 会带来 STW 暂停和并发标记期间的调度影响，可能让请求延迟出现尖刺。

表现：

- P99、P999 延迟升高。
- 偶发超时。
- 请求耗时分布长尾明显。

### 2. CPU 占用升高

GC 并发标记需要 CPU。分配越快，GC 工作越重，业务可用 CPU 越少。

表现：

- CPU 使用率升高但业务吞吐不涨。
- 火焰图里 runtime GC 相关函数占比变高。
- 高峰期频繁 GC。

### 3. 吞吐下降

当 GC 占用 CPU 或导致频繁内存分配时，业务处理能力会下降。

### 4. 内存占用波动

Go GC 通常允许堆增长到一定比例后再触发回收。分配速率高时，堆可能快速增长。

### 5. 调度压力

大量 Goroutine、大量小对象、复杂指针图会增加扫描成本，间接影响调度。

## 哪些代码会加重 GC 压力

### 1. 高频创建临时对象

```go
for _, item := range items {
    data := make([]byte, 1024)
    process(data, item)
}
```

如果在高 QPS 路径中频繁分配临时对象，会显著增加分配速率。

### 2. 字符串和 []byte 频繁转换

```go
s := string(b)
b2 := []byte(s)
```

这类转换通常会产生拷贝和新对象。

### 3. 使用 interface{} 导致装箱和逃逸

```go
func logAny(v any) {
    fmt.Println(v)
}
```

不一定总是严重，但在热点路径中要关注逃逸和分配。

### 4. 大量指针对象

指针越多，GC 需要扫描的引用关系越多。

例如：

```go
[]*Item
```

相比：

```go
[]Item
```

前者可能增加 GC 扫描压力，但后者可能增加拷贝成本，需要权衡。

### 5. 无上限缓存

缓存一直增长，会让堆变大，GC 扫描成本也变大。

## 常见优化方向

### 1. 减少分配

预分配 slice：

```go
result := make([]Item, 0, len(input))
```

避免在循环中重复创建对象。

### 2. 复用对象

可以使用 `sync.Pool` 复用临时对象：

```go
var bufPool = sync.Pool{
    New: func() any {
        return new(bytes.Buffer)
    },
}

buf := bufPool.Get().(*bytes.Buffer)
buf.Reset()
defer bufPool.Put(buf)
```

注意：`sync.Pool` 适合临时对象缓存，不适合作为可靠对象池，因为 GC 可能清理 pool 中的对象。

### 3. 降低指针数量

尽量使用连续内存和非指针字段，减少 GC 扫描量。

例如在性能敏感路径中，可以考虑：

- `[]Struct` 而不是 `[]*Struct`。
- 小对象合并。
- 避免过度链表结构。

### 4. 控制缓存大小

缓存要有：

- 最大容量。
- TTL。
- 淘汰策略。
- 指标监控。

### 5. 避免内存泄漏

泄漏会让存活对象越来越多，GC 成本越来越高。

典型泄漏：

- Goroutine 泄漏。
- channel 阻塞。
- map/cache 无上限。
- slice 小片引用大数组。
- timer/ticker 未停止。

### 6. 调整 GOGC

`GOGC` 控制下一次 GC 触发时堆相对上次存活堆的增长比例。

默认值通常是 `100`，表示堆大约增长到上次存活堆的 2 倍时触发下一轮 GC。

调低 GOGC：

- GC 更频繁。
- 内存占用更低。
- CPU 开销可能更高。

调高 GOGC：

- GC 较少。
- 内存占用更高。
- CPU 开销可能更低，但单次扫描堆可能更大。

要通过压测和监控决定，不要盲调。

## 排查工具

### pprof heap

查看堆内存分配和存活对象。

```bash
go tool pprof http://localhost:6060/debug/pprof/heap
```

重点看：

- 哪些函数分配最多。
- 哪些对象长期存活。
- inuse_space 和 alloc_space。

### pprof allocs

查看累计分配。

```bash
go tool pprof http://localhost:6060/debug/pprof/allocs
```

### go test -bench -benchmem

看每次操作分配次数和分配字节数。

```bash
go test -bench=. -benchmem
```

### GODEBUG=gctrace=1

输出 GC 日志。

```bash
GODEBUG=gctrace=1 ./app
```

可以观察 GC 频率、堆大小、暂停时间等。

### runtime/metrics

适合在服务中采集 GC 指标。

## 面试回答模板

Go GC 是并发标记清扫，大部分标记工作和用户程序并发执行，STW 时间通常较短。但在高并发系统中，GC 仍然会影响 P99 延迟、CPU 使用率和吞吐。分配速率越高、堆越大、指针越多，GC 压力越明显。

我优化 GC 问题时不会先想着关 GC，而是先用 pprof、gctrace、benchmark 找到分配热点。常见手段包括 slice 预分配、减少临时对象、复用 buffer、控制缓存大小、避免 Goroutine 和 timer 泄漏、减少不必要指针。对于确实需要权衡内存和 CPU 的场景，再结合压测调整 GOGC。

## 常见追问

### 1. Go GC 会不会导致长时间 STW？

现代 Go 的 GC 目标是降低 STW，大部分工作并发完成。但 STW 不是完全没有，低延迟系统仍需关注 P99/P999 和 GC 指标。

### 2. sync.Pool 能解决所有 GC 问题吗？

不能。`sync.Pool` 适合复用临时对象，降低分配压力。但对象可能被 GC 清理，不能当作可靠缓存。滥用还可能增加复杂度。

### 3. GOGC 调大一定更好吗？

不一定。调大 GOGC 可能降低 GC 频率和 CPU 开销，但会增加内存占用，也可能让单次扫描堆变大。要根据服务目标和压测结果调整。

### 4. 如何判断 GC 是瓶颈？

看现象和数据：P99 延迟是否和 GC 时间相关，CPU 中 runtime GC 占比是否高，分配速率是否大，heap 是否持续增长，gctrace 是否显示频繁 GC。最终要结合 pprof 和 trace 定位。
