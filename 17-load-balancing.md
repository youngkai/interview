# 负载均衡

## 30 秒回答

负载均衡是把请求分发到多个服务实例上，提高吞吐、可用性和扩展能力。常见层次包括 DNS 负载均衡、四层负载均衡、七层负载均衡、服务发现客户端负载均衡等。

跨境物流系统中，订单查询、轨迹查询、承运商回调、开放平台 API 都可以通过负载均衡分摊请求。设计时要关注负载均衡算法、健康检查、会话保持、超时、重试、灰度发布和故障摘除。

## 负载均衡的层次

### 1. DNS 负载均衡

通过 DNS 返回多个 IP。

优点：

- 简单。
- 适合跨地域入口。

缺点：

- 生效慢。
- 健康检查能力有限。
- 客户端和运营商缓存可能导致流量切换不及时。

### 2. 四层负载均衡

工作在 TCP/UDP 层，根据 IP 和端口转发。

特点：

- 性能高。
- 不理解 HTTP 内容。
- 常用于高吞吐入口。

### 3. 七层负载均衡

工作在 HTTP 层，可以根据 path、header、host 等转发。

示例：

```text
/api/orders    -> order-service
/api/tracking  -> tracking-service
/api/inventory -> inventory-service
```

优点：

- 路由灵活。
- 支持灰度、鉴权、限流、日志。

缺点：

- 性能相对四层更低。
- 规则复杂后维护成本增加。

### 4. 客户端负载均衡

服务调用方从注册中心拿到实例列表，在客户端选择实例。

常见于微服务内部调用。

## 常见算法

### 1. 轮询

请求按顺序分发到每个实例。

适合实例配置差不多的场景。

### 2. 加权轮询

配置高的机器权重大，承担更多流量。

### 3. 随机

随机选择实例，简单直接。

### 4. 最少连接

把请求发给当前连接数最少的实例。

适合请求耗时差异较大的场景。

### 5. 一致性哈希

同一个 key 尽量路由到同一个节点。

适合：

- 缓存节点。
- 需要会话亲和的场景。
- 降低节点变更时的迁移范围。

## 健康检查

负载均衡必须能发现坏实例并摘除。

健康检查包括：

- 进程是否存活。
- HTTP health endpoint 是否正常。
- 依赖是否可用。
- 延迟是否过高。

简单健康检查：

```http
GET /healthz
```

更细粒度：

```http
GET /readyz
```

`healthz` 表示进程活着，`readyz` 表示实例可以接收流量。

## 会话保持

如果服务是无状态的，一般不需要会话保持。

如果必须让同一用户路由到同一实例，可以用：

- cookie。
- header。
- source IP。
- consistent hash。

但会话保持会影响扩缩容和故障切换，所以更推荐服务无状态化，把状态放到 Redis、数据库或 token 中。

## 负载均衡和重试

负载均衡层或客户端可能在失败时重试其他实例。

风险：

- 非幂等请求被重复执行。
- 下游压力被放大。
- 超时设置不合理导致请求堆积。

所以重试要配合：

- 幂等性。
- 超时。
- 最大重试次数。
- 退避。
- 只对可重试错误重试。

## 面试回答模板

负载均衡的作用是把请求分散到多个实例上，提高系统吞吐和可用性。入口层可以用四层或七层负载均衡，七层可以根据 path、header 做路由，也方便做限流、灰度和日志。微服务内部可以通过服务发现和客户端负载均衡选择实例。

算法上常见有轮询、加权轮询、随机、最少连接、一致性哈希。服务实例必须有健康检查，异常实例要及时摘除。对于跨境物流这种多服务系统，我会尽量让服务无状态化，方便负载均衡和水平扩容。

需要注意的是，负载均衡重试可能导致非幂等请求重复执行，所以订单创建、库存扣减、支付这类接口必须有幂等设计。

## 常见追问

### 1. 四层和七层负载均衡区别是什么？

四层基于 TCP/UDP 转发，性能高但不理解业务请求。七层基于 HTTP，可以根据 URL、header、cookie 路由，功能更丰富。

### 2. 为什么要健康检查？

没有健康检查，负载均衡可能继续把请求发到异常实例，导致错误率升高。

### 3. 一致性哈希解决什么问题？

节点增加或减少时，一致性哈希能减少 key 迁移范围，常用于缓存负载均衡。
