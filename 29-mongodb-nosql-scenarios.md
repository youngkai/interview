# MongoDB 等 NoSQL 使用场景

## 30 秒回答

NoSQL 适合处理关系不强、结构灵活、写入量大或需要横向扩展的数据。MongoDB 是文档数据库，适合存半结构化 JSON 文档，例如物流轨迹原始事件、承运商回调报文、订单扩展字段、清关附加信息、操作日志等。

但核心交易数据，比如订单主状态、库存数量、支付流水，通常仍更适合关系型数据库，因为它们需要事务、约束、强一致和复杂查询。NoSQL 的关键是选对场景，不是用它替代所有数据库。

## MongoDB 的特点

### 1. 文档模型

数据以 BSON 文档存储，结构类似 JSON。

示例：

```json
{
  "tracking_no": "YT123",
  "carrier": "carrier_a",
  "events": [
    {
      "status": "CUSTOMS_RELEASED",
      "event_time": "2026-05-16T10:00:00Z",
      "raw": {}
    }
  ]
}
```

适合结构变化较多的数据。

### 2. 灵活 schema

不同承运商回调字段可能不一样，MongoDB 可以更自然地保存原始数据。

### 3. 横向扩展

MongoDB 支持分片，适合大数据量文档存储。

### 4. 索引能力

支持单字段、复合、多键、TTL、文本等索引。

## 适合 MongoDB 的跨境物流场景

### 1. 物流轨迹事件

轨迹事件量大，字段可能因承运商不同而变化。

可以按 tracking_no 存文档，或者每个事件一条文档。

设计方式一：一个包裹一个文档。

优点：

- 查询一个包裹轨迹方便。

缺点：

- events 数组过大时文档膨胀。
- 并发追加同一文档可能有热点。

设计方式二：一个轨迹事件一条文档。

优点：

- 写入扩展性更好。
- 单条事件更小。

缺点：

- 查询完整轨迹需要按 tracking_no 聚合。

### 2. 承运商原始回调

不同承运商字段差异很大，保存原始报文便于排查。

```json
{
  "carrier": "DHL",
  "tracking_no": "xxx",
  "received_at": "2026-05-16T10:00:00Z",
  "raw_payload": {}
}
```

### 3. 订单扩展字段

不同国家、不同清关要求有不同扩展信息。

但核心字段仍应放关系型数据库：

- order_id。
- tenant_id。
- status。
- amount。
- created_at。

扩展字段可以放 JSONB 或 MongoDB。

### 4. 操作日志和审计日志

日志类数据写多读少、结构可能变化，适合 NoSQL 或日志系统。

### 5. 内容和配置类数据

某些复杂配置文档可以存 MongoDB，但高频读取仍可加 Redis 缓存。

## 不适合 MongoDB 的场景

### 1. 强事务核心链路

例如：

- 库存扣减。
- 支付扣款。
- 订单主状态严谨流转。
- 财务结算。

这些更适合关系型数据库。

### 2. 复杂多表关联

MongoDB 不适合大量复杂 join。关系清晰的数据更适合 RDBMS。

### 3. 高度规范化数据

如果数据天然有强关系、强约束，关系型数据库更自然。

## MongoDB 索引设计

按查询建索引。

轨迹查询：

```text
(tracking_no, event_time)
```

承运商回调排查：

```text
(carrier, received_at)
```

租户维度查询：

```text
(tenant_id, created_at)
```

TTL 索引适合自动清理临时数据：

```text
expire_at
```

注意：索引太多会影响写入性能。

## MongoDB 数据建模原则

### 1. 根据查询模式建模

不要只按对象关系建模，要看最常见查询是什么。

### 2. 能嵌入就嵌入，但避免无限增长数组

一个订单的少量地址信息可以嵌入。一个包裹的无限轨迹事件不一定适合全部嵌在一个文档里。

### 3. 热点文档要避免频繁更新

如果大量事件都追加到同一个文档，可能形成写热点。

### 4. 核心查询字段冗余出来

即使原始 payload 放 raw，也要把常用查询字段提取成顶层字段：

```text
tracking_no
carrier
event_time
status
tenant_id
```

这样才能建索引。

## NoSQL 选型简述

| 类型 | 代表 | 适合场景 |
| --- | --- | --- |
| 文档数据库 | MongoDB | 半结构化文档、灵活 schema |
| KV 数据库 | Redis | 缓存、计数、限流、锁 |
| 宽列数据库 | Cassandra/HBase | 海量写入、时间序列、日志 |
| 搜索引擎 | Elasticsearch/OpenSearch | 全文检索、多条件搜索、聚合 |
| 时序数据库 | InfluxDB/TimescaleDB | 指标、时间序列 |

## 面试回答模板

MongoDB 适合存文档型、半结构化、字段变化多的数据。跨境物流里，不同承运商回调格式差异大，物流轨迹原始事件、清关扩展信息、操作日志都可以放 MongoDB。常用查询字段要提取出来建索引，比如 tracking_no、carrier、event_time、tenant_id。

但我不会把订单主表、库存扣减、支付流水这类核心交易数据随意放 MongoDB。它们需要事务、唯一约束、状态机和强一致，更适合 MySQL/PostgreSQL。NoSQL 是补充，不是替代所有关系型数据库。

## 常见追问

### 1. 物流轨迹应该一个包裹一个文档，还是一个事件一个文档？

看事件数量和写入模式。事件少、查询整条轨迹多，可以一个包裹一个文档；事件多、追加频繁、高写入量，更适合一个事件一条文档，按 tracking_no 查询聚合。

### 2. MongoDB 需要索引吗？

需要。NoSQL 不代表不用索引。高频查询字段必须建索引，否则数据量大时一样慢。

### 3. NoSQL 能保证一致性吗？

不同 NoSQL 能力不同。核心业务不要只依赖模糊的一致性承诺，要结合具体数据库特性、写入确认、事务能力和业务补偿设计。
