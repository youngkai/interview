# 数据库分库分表策略

## 30 秒回答

分库分表是为了解决单库单表在数据量、写入吞吐、索引大小、连接数和存储容量上的瓶颈。跨境物流里，订单表、轨迹事件表、库存流水表、操作日志表都可能随着业务增长变得很大。

分库分表要先选好分片键，常见有 `order_id`、`user_id`、`tenant_id`、`tracking_no`。设计时要考虑查询模式、扩容迁移、全局唯一 ID、跨分片查询、分布式事务、热点分片和历史数据归档。分库分表不是第一选择，只有单库优化、索引、读写分离、归档后仍不够时再做。

## 什么时候需要分库分表

常见信号：

- 单表数据量过大，索引膨胀明显。
- 写入 QPS 接近单库瓶颈。
- 查询变慢且优化空间有限。
- 单库连接数和 CPU 压力过高。
- 单机存储容量接近上限。
- 备份恢复时间过长。

不要一开始就分库分表，因为它会显著增加系统复杂度。

## 垂直拆分和水平拆分

### 垂直拆分

按业务模块拆库：

```text
order_db
inventory_db
shipment_db
tracking_db
payment_db
```

适合微服务按业务边界拆分。

### 水平拆分

同一张表按规则拆到多个表或多个库。

例如订单表：

```text
order_00
order_01
...
order_63
```

按 `order_id % 64` 分表。

## 分片键选择

分片键非常关键，选错会导致大量跨分片查询。

### order_id

适合订单详情查询、订单状态更新。

优点：

- 订单主链路天然有 order_id。
- 单订单操作定位准确。

缺点：

- 按用户查订单需要额外索引或冗余表。

### user_id / tenant_id

适合按用户或商家查询订单列表。

优点：

- 用户维度查询方便。
- 多租户隔离更好。

缺点：

- 大商家可能形成热点。
- 单个订单操作如果只有 order_id，需要先路由。

### tracking_no

适合物流轨迹表。

优点：

- 查询轨迹通常按 tracking_no。
- 同一包裹事件可以落在同一分片。

缺点：

- 按订单聚合多个包裹时需要额外关联。

## 常见分片策略

### 1. 哈希分片

```text
shard = hash(order_id) % N
```

优点：

- 数据分布均匀。
- 实现简单。

缺点：

- 扩容迁移麻烦。
- 范围查询不友好。

### 2. 范围分片

按时间或 ID 范围：

```text
order_2026_01
order_2026_02
```

优点：

- 适合时间范围查询和归档。
- 历史数据管理方便。

缺点：

- 当前时间分片可能成为写热点。

### 3. 一致性哈希

减少扩容时的数据迁移范围。

更常用于缓存或中间件路由，数据库场景也可以参考。

### 4. 混合分片

例如先按租户分库，再按订单 ID 分表。

适合大规模 SaaS 或多租户系统，但复杂度更高。

## 全局唯一 ID

分库分表后不能依赖单库自增 ID。

常见方案：

- 雪花算法。
- 数据库号段。
- UUID/ULID。
- Redis/incr 号段。

订单 ID 最好包含时间趋势，方便排序和排查，但也要避免单调递增导致热点。

## 跨分片查询问题

例如按用户查询最近订单，但订单按 order_id 分片。

解决：

- 建用户订单关系表，按 user_id 分片。
- 写入时冗余查询维度。
- 使用搜索引擎。
- 使用数据仓库做分析查询。
- 限制查询条件，避免全分片扫描。

## 分布式事务问题

分库分表后，一个业务操作可能涉及多个库。

解决：

- 尽量让一次业务操作落在一个分片。
- 使用最终一致性。
- 使用 TCC/Saga。
- 用消息驱动状态同步。

## 热点问题

如果按 tenant_id 分片，大商家可能集中打到一个分片。

解决：

- 大租户单独拆分。
- tenant_id + order_id 做复合分片。
- 热点租户独立库。
- 读请求加缓存。

## 历史数据归档

跨境物流订单和轨迹数据会长期增长。要有归档策略：

- 热数据保留在在线库。
- 冷数据进入归档库或对象存储。
- 查询时区分近期和历史。
- 报表走 OLAP，不压在线库。

## 面试回答模板

分库分表是为了解决单库单表的数据量和吞吐瓶颈，但它会带来跨分片查询、分布式事务、扩容迁移和全局 ID 问题，所以不是第一步就做。一般会先做索引优化、缓存、读写分离、归档和 SQL 优化。

如果订单表确实很大，我会根据主要查询模式选择分片键。订单详情和状态更新通常按 order_id 查询，可以按 order_id 哈希分片；如果商家订单列表是核心查询，就需要按 tenant_id 或建立 tenant_id 维度的冗余索引表。物流轨迹表可以按 tracking_no 分片，保证同一个包裹的轨迹事件集中在一个分片。

分库分表后要解决全局唯一 ID、跨分片查询、热点租户、扩容迁移和最终一致性。分析报表不要直接扫在线分片库，应该同步到搜索或数仓。

## 常见追问

### 1. 分片键怎么选？

看最核心、最高频的查询条件。能让大多数请求定位到单个分片的字段，才适合作为分片键。

### 2. 分库分表后如何分页？

单分片分页比较简单。跨分片分页很复杂，通常要限制查询维度，或者借助搜索引擎、冗余表、游标分页和排序字段。

### 3. 分库分表后如何扩容？

可以提前预分片，也可以通过中间件迁移。扩容要考虑数据搬迁、双写、校验、灰度切流和回滚方案。
