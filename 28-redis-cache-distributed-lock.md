# Redis 缓存策略和分布式锁

## 30 秒回答

Redis 常用于缓存热点数据、计数器、限流、分布式锁、延迟队列和会话状态。跨境物流里可以缓存订单摘要、物流轨迹最新状态、国家地区配置、承运商配置、运费模板等读多写少数据。

缓存策略要处理一致性、穿透、击穿、雪崩、热点 key 和大 key。分布式锁可以用 `SET key value NX EX seconds` 实现，但必须设置过期时间、释放时校验 value，并理解它不能替代数据库事务。库存扣减、订单状态更新这类核心一致性最终仍要靠数据库条件更新、唯一约束和幂等设计兜底。

## Redis 常见使用场景

### 1. 缓存

适合：

- 订单查询摘要。
- 物流轨迹最新节点。
- 承运商配置。
- 国家地区字典。
- 运费模板。
- 用户权限摘要。

### 2. 计数器

例如接口调用次数、限流计数、库存预热数量。

```text
INCR api:tenant:1001:count
```

### 3. 分布式锁

协调多个服务实例对同一资源的操作。

### 4. 排行榜和有序任务

ZSet 可用于排行榜、延迟队列。

### 5. 去重

使用 SET 或 String 记录 request id、event id。

## 缓存模式

### Cache Aside

最常见。

读流程：

```text
读缓存
命中 -> 返回
未命中 -> 查数据库 -> 写缓存 -> 返回
```

写流程：

```text
更新数据库 -> 删除缓存
```

为什么通常删除缓存？

因为并发更新缓存容易出现旧值覆盖新值。删除缓存让下一次读重新加载数据库最新值，逻辑更稳。

## 缓存一致性

缓存和数据库很难做到强一致，通常做最终一致。

常见做法：

- 更新数据库后删除缓存。
- 缓存设置 TTL。
- 删除缓存失败要重试。
- 通过 MQ/CDC 通知删除缓存。
- 对核心读请求必要时读数据库。

注意：订单支付状态、库存最终数量不能只相信缓存。

## 缓存穿透

查询不存在的数据，每次都打到数据库。

解决：

- 缓存空值，设置短 TTL。
- 布隆过滤器。
- 参数校验。
- 黑名单。
- 限流。

例子：大量查询不存在的 tracking_no。

## 缓存击穿

热点 key 过期瞬间，大量请求同时查数据库。

解决：

- singleflight 合并请求。
- 互斥锁重建缓存。
- 热点 key 不过期，后台刷新。
- 提前异步刷新。

## 缓存雪崩

大量 key 同时过期或 Redis 故障，流量集中打数据库。

解决：

- TTL 加随机抖动。
- 多级缓存。
- Redis 高可用。
- 限流降级。
- 热点数据预热。

## 热点 key 和大 key

### 热点 key

某个订单、轨迹或商品被大量访问。

解决：

- 本地缓存。
- 多副本缓存。
- 请求合并。
- 限流。
- key 拆分。

### 大 key

value 太大或集合元素太多。

风险：

- 网络传输慢。
- Redis 阻塞。
- 删除慢。
- 内存碎片。

解决：

- 拆分 key。
- 分页读取。
- 控制集合大小。
- 异步删除。

## 分布式锁基础

简单写法：

```text
SET lock:order:1001 request-id NX EX 10
```

含义：

- `NX`：不存在才设置，保证互斥。
- `EX 10`：设置过期时间，防止死锁。
- value：唯一请求 ID，用于释放锁时校验持有者。

释放锁不能直接 `DEL`，要先判断 value 是不是自己。

Lua 释放：

```lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("DEL", KEYS[1])
end
return 0
```

## 分布式锁常见坑

### 1. 没有过期时间

持锁进程崩溃后，锁永远不释放。

### 2. 释放了别人的锁

请求 A 加锁后阻塞超过过期时间，锁被请求 B 获取。请求 A 恢复后直接 DEL，会删掉 B 的锁。

所以释放时必须校验 value。

### 3. 锁过期但业务没执行完

业务执行时间超过锁 TTL，锁提前释放，其他请求进入临界区。

解决：

- 合理设置 TTL。
- 锁续期。
- 缩短临界区。
- 最终仍靠数据库约束兜底。

### 4. 把分布式锁当成强一致事务

Redis 锁可以减少并发冲突，但不能替代数据库事务、唯一约束和状态机。

例如防重复创建订单，不能只靠 Redis 锁，还要有数据库唯一索引。

## 分布式锁适合场景

适合：

- 防止缓存击穿时多实例同时重建缓存。
- 定时任务多实例抢占执行权。
- 同一订单短时间内只允许一个实例处理。
- 非强金融一致性的资源互斥。

不适合单独承担：

- 支付扣款最终一致。
- 库存最终扣减。
- 订单唯一性。

这些必须有数据库或业务状态兜底。

## 面试回答模板

Redis 在跨境物流系统里主要用于缓存热点读、限流计数、去重、延迟队列和分布式锁。缓存上我会用 Cache Aside，读缓存 miss 后查库再写缓存，写数据时先更新数据库再删除缓存，并设置 TTL。对于穿透、击穿、雪崩，分别用缓存空值/布隆过滤器、singleflight 或互斥锁、TTL 随机抖动和限流降级处理。

分布式锁可以用 `SET key value NX EX`，value 必须是唯一 token，释放时用 Lua 校验 value 后删除。锁必须有过期时间，业务执行太久要考虑续期。但我不会只依赖 Redis 锁保证核心一致性，订单、库存、支付最终还是要靠数据库条件更新、唯一索引、状态机和幂等。

## 常见追问

### 1. Redis 锁为什么要设置 value？

为了释放锁时确认锁还是自己持有的，避免误删其他请求新获取的锁。

### 2. Redis 缓存和数据库怎么保证一致？

通常保证最终一致。更新数据库后删除缓存，删除失败重试，缓存设置 TTL，必要时通过 MQ 或 CDC 删除缓存。

### 3. Redis 做库存扣减安全吗？

Redis Lua 可以保证单 Redis 节点上的原子扣减，适合秒杀前置削峰。但数据库或库存系统仍要有最终扣减、流水、对账和补偿，不能只依赖 Redis。
