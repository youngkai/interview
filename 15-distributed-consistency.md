# 数据一致性问题：分布式事务、最终一致性

## 30 秒回答

微服务中每个服务通常有自己的数据库，跨服务操作很难用本地事务一次性保证强一致。比如下单需要创建订单、预占库存、创建支付单、通知仓储，这些动作分布在不同服务里。

常见方案有分布式事务、TCC、Saga、本地消息表、事务消息和最终一致性。实际业务中，强一致成本很高，更多会采用最终一致性，通过状态机、消息队列、幂等、重试、补偿、对账来保证最终结果正确。

## 为什么会有一致性问题

单体系统里，订单和库存可能在同一个数据库事务中完成：

```text
begin
  insert order
  update stock
commit
```

微服务后：

```text
订单服务 -> 订单库
库存服务 -> 库存库
支付服务 -> 支付库
```

一次业务操作跨多个服务和多个数据库，本地事务无法覆盖所有资源。

## 常见一致性模型

### 1. 强一致性

操作完成后，所有系统立刻看到一致结果。

优点：

- 业务语义简单。
- 用户体验确定。

缺点：

- 分布式环境成本高。
- 延迟高。
- 可用性受影响。

### 2. 最终一致性

短时间内允许中间状态不一致，但系统通过消息、重试、补偿、对账，最终达到一致。

跨境物流更常见最终一致性，因为链路长、外部系统多、运输和清关本身就是异步过程。

## 常见方案

### 1. 2PC

两阶段提交：

1. prepare 阶段：协调者询问所有参与者能否提交。
2. commit 阶段：所有参与者提交或回滚。

缺点：

- 协调者单点风险。
- 参与者阻塞。
- 性能差。
- 不适合高并发互联网业务。

面试中可以说了解，但实际业务一般慎用。

### 2. TCC

TCC 分为：

- Try：预留资源。
- Confirm：确认提交。
- Cancel：取消释放。

库存场景很典型：

```text
Try: 预占库存
Confirm: 扣减预占库存
Cancel: 释放预占库存
```

优点：

- 业务可控。
- 一致性较强。

缺点：

- 业务侵入大。
- 每个资源都要实现 Try/Confirm/Cancel。
- 幂等、空回滚、悬挂问题复杂。

### 3. Saga

Saga 把一个大事务拆成多个本地事务，每个本地事务都有对应补偿动作。

示例：

```text
创建订单 -> 预占库存 -> 创建支付单 -> 通知仓储
```

如果创建支付单失败：

```text
释放库存 -> 取消订单
```

Saga 适合长流程业务，比如订单履约、跨境运输、退款退货。

### 4. 本地消息表

在本地事务里同时写业务数据和消息表。

```text
begin
  insert order
  insert outbox_message
commit
```

后台任务扫描消息表，把消息发送到 MQ。发送成功后标记消息已发送。

优点：

- 保证业务数据和待发送消息在同一个本地事务中提交。
- 实现相对可控。

缺点：

- 需要扫描或 CDC。
- 消息可能重复投递，消费者必须幂等。

### 5. 事务消息

某些 MQ 支持事务消息。大致流程：

1. 先发送半消息。
2. 执行业务本地事务。
3. 提交或回滚消息。
4. MQ 可以回查事务状态。

适合订单创建后通知下游处理。

### 6. 对账补偿

即使有消息和事务机制，也需要定时对账。

例如：

- 订单已支付但库存未确认。
- 运单已创建但订单状态未更新。
- 包裹已签收但订单仍显示运输中。

对账任务发现异常后重新投递事件或执行补偿。

## 幂等是最终一致的基础

消息会重复，接口会重试，所以消费者必须幂等。

常见方式：

- 消息唯一 ID。
- 业务唯一键。
- 去重表。
- 状态机判断。
- 数据库唯一索引。

示例：

```text
event_id = shipment-1001-delivered-001
```

消费者处理前先判断是否已处理。

## 状态机防乱序

跨境物流状态可能乱序到达：

```text
DELIVERED 先到
CUSTOMS_RELEASED 后到
```

订单或轨迹服务不能简单用新消息覆盖旧状态，要基于状态机和事件时间处理。

可以做：

- 状态流转校验。
- 事件版本号。
- 事件发生时间。
- 幂等事件表。
- 异常状态进入人工或补偿队列。

## 面试回答模板

微服务下每个服务有自己的数据库，跨订单、库存、支付、运输这些服务时，本地事务无法保证整体强一致。强一致方案比如 2PC 成本高、可用性差，互联网业务里通常更倾向最终一致性。

我会根据业务选择 TCC、Saga、本地消息表或事务消息。比如库存扣减可以用 TCC，先预占库存，支付成功后确认扣减，支付失败或超时释放库存。订单履约这种长链路更适合 Saga，每一步是本地事务，失败后执行补偿。

最终一致性的关键是消息可靠投递、消费者幂等、状态机防乱序、失败重试和定时对账。跨境物流外部系统多，不能假设所有状态实时一致，必须允许中间态，并通过补偿机制保证最终正确。

## 常见追问

### 1. 最终一致是不是一定会丢数据？

不是。最终一致不是降低可靠性，而是允许短暂不一致。通过可靠消息、幂等、重试、补偿和对账，可以保证最终结果正确。

### 2. TCC 的难点是什么？

难点是业务侵入大，还要处理幂等、空回滚、悬挂。比如 Cancel 先于 Try 到达，或者 Confirm 重复执行，都要正确处理。

### 3. 消息发送成功但本地事务失败怎么办？

要避免这种问题，可以使用本地消息表或事务消息，让业务变更和消息发送具备一致性保障。
