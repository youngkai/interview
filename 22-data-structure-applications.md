# 栈、队列、链表、哈希表实际应用场景

## 30 秒回答

栈、队列、链表、哈希表是工程里最常见的基础数据结构。栈适合处理“后进先出”的场景，例如函数调用、表达式解析、撤销操作；队列适合“先进先出”的任务排队，例如消息队列、请求缓冲、生产者消费者；链表适合频繁插入删除和顺序连接，例如 LRU 缓存；哈希表适合快速查找，例如缓存、索引、去重和映射关系。

面试时不要只背定义，要能把它们和真实系统联系起来，比如订单系统里的幂等去重用哈希表，库存扣减异步处理用队列，LRU 缓存用哈希表加双向链表。

## 栈

### 特点

栈是后进先出结构：

```text
push -> 入栈
pop  -> 出栈
top  -> 查看栈顶
```

最后进入的数据最先出来。

## 栈的应用场景

### 1. 函数调用栈

程序执行函数调用时，运行时会维护调用栈。每次函数调用入栈，函数返回出栈。

递归本质上也依赖调用栈：

```go
func factorial(n int) int {
    if n <= 1 {
        return 1
    }
    return n * factorial(n-1)
}
```

### 2. 表达式解析

例如判断括号是否合法：

```text
{[()]}  合法
{[(])}  不合法
```

遇到左括号入栈，遇到右括号和栈顶匹配。

### 3. 撤销和回退

编辑器的 undo、浏览器后退、操作历史都可以用栈。

```text
操作 A -> 操作 B -> 操作 C
undo 时先撤销 C
```

### 4. 深度优先搜索

DFS 可以用递归，也可以显式使用栈实现。

## 队列

### 特点

队列是先进先出结构：

```text
enqueue -> 入队
dequeue -> 出队
```

最先进入的数据最先被处理。

## 队列的应用场景

### 1. 生产者消费者

生产者把任务放进队列，消费者从队列取任务处理。

Go 中可以直接用 channel 表达：

```go
jobs := make(chan Job, 100)
```

### 2. 消息队列

Kafka、RabbitMQ、RocketMQ 都可以看作工程化的队列系统，用于：

- 异步解耦。
- 削峰填谷。
- 最终一致性。
- 任务重试。

### 3. 请求排队

高并发服务中，可以使用有界队列缓冲请求。队列满时快速失败，防止内存无限增长。

### 4. 广度优先搜索

BFS 使用队列，常见于最短路径、层序遍历。

### 5. 延迟队列

订单超时未支付自动取消，可以用延迟队列。

```text
创建订单 -> 延迟 15 分钟消息 -> 检查是否支付 -> 未支付则取消
```

## 链表

### 特点

链表由节点组成，每个节点保存数据和指向其他节点的指针。

单向链表：

```text
A -> B -> C
```

双向链表：

```text
A <-> B <-> C
```

优点：

- 插入删除不需要移动大量元素。
- 节点可以灵活连接。

缺点：

- 不支持 O(1) 随机访问。
- 指针多，缓存局部性不如数组。

## 链表的应用场景

### 1. LRU 缓存

LRU 通常用哈希表加双向链表：

- 哈希表负责 O(1) 查找 key。
- 双向链表维护访问顺序。
- 最近访问的放到链表头。
- 淘汰链表尾部最久未使用节点。

### 2. 操作系统调度队列

很多调度器会用链表组织任务。

### 3. 内存管理

空闲内存块可以通过链表组织。

### 4. 频繁插入删除的列表

如果数据量大且经常在中间插入删除，链表比数组少了搬移元素的成本。

## 哈希表

### 特点

哈希表通过 hash 函数把 key 映射到数组位置，实现快速查找。

平均时间复杂度：

- 查找：O(1)
- 插入：O(1)
- 删除：O(1)

最坏情况下可能退化，但工程实现会通过扩容、冲突处理等方式降低概率。

## 哈希表的应用场景

### 1. 缓存

```go
cache := map[string]User{}
```

通过 key 快速获取 value。

### 2. 去重

```go
seen := map[string]struct{}{}
```

常用于消息幂等、订单去重、重复请求过滤。

### 3. 索引

把某个字段映射到对象：

```go
orderByID := map[int64]*Order{}
```

### 4. 计数器

统计词频、接口调用次数、错误码数量：

```go
counter := map[string]int{}
```

### 5. 分布式路由

一致性哈希用 hash 决定 key 落在哪个节点。

## 组合应用：LRU 缓存

LRU 是典型组合题。

只用链表：

- 可以维护访问顺序。
- 但查找 key 是 O(n)。

只用哈希表：

- 查找是 O(1)。
- 但不知道谁最久没访问。

组合后：

```text
map[key]*node + doubly linked list
```

可以做到：

- Get：O(1)
- Put：O(1)
- 淘汰：O(1)

## 面试回答模板

栈适合后进先出的场景，比如函数调用、括号匹配、表达式解析、撤销操作和 DFS。队列适合先进先出的场景，比如生产者消费者、消息队列、请求排队、BFS 和延迟任务。

链表适合频繁插入删除，比如 LRU 缓存的访问顺序维护。哈希表适合快速查找、去重、缓存、计数和索引。工程里很多场景不是单独用一个结构，而是组合使用。比如 LRU 缓存就是哈希表加双向链表，哈希表负责 O(1) 查找，链表负责 O(1) 移动和淘汰。

## 常见追问

### 1. 为什么 LRU 要用双向链表？

因为访问一个节点后要把它移动到头部，淘汰时要删除尾部节点。双向链表能在已知节点指针的情况下 O(1) 删除和移动。

### 2. 哈希表一定是 O(1) 吗？

平均是 O(1)，但不是绝对。哈希冲突严重时会退化，实际实现会通过扩容和冲突处理降低风险。

### 3. 数组和链表怎么选？

数组随机访问快、缓存局部性好；链表插入删除灵活，但查找慢、指针开销大。实际选择看访问模式。
