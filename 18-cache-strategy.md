# 缓存策略：Redis 和 Memcached

## 30 秒回答

缓存用于提升读性能、降低数据库压力。跨境物流里适合缓存的有物流轨迹最新状态、国家地区配置、承运商配置、运费模板、商品基础信息等。

Redis 功能更丰富，支持多种数据结构、持久化、Lua、分布式锁、Stream 等；Memcached 更简单，主要是高性能 key-value 缓存。实际业务中 Redis 使用更广。缓存设计要重点关注一致性、穿透、击穿、雪崩、热点 key、大 key 和过期策略。

## Redis vs Memcached

| 对比项 | Redis | Memcached |
| --- | --- | --- |
| 数据结构 | string、hash、list、set、zset、stream 等 | 简单 key-value |
| 持久化 | 支持 RDB/AOF | 不支持或很弱 |
| 高可用 | Sentinel、Cluster | 依赖客户端和部署方案 |
| 原子操作 | 丰富 | 较少 |
| Lua 脚本 | 支持 | 不支持 |
| 使用场景 | 缓存、计数、排行榜、分布式锁、队列等 | 简单缓存 |

## 常见缓存模式

### 1. Cache Aside

最常见模式，由业务代码管理缓存。

读流程：

```text
先读缓存
缓存命中 -> 返回
缓存未命中 -> 查数据库 -> 写缓存 -> 返回
```

写流程：

```text
更新数据库 -> 删除缓存
```

为什么通常是删除缓存而不是更新缓存？

因为更新缓存容易出现并发写入导致旧值覆盖新值。删除缓存后，下一次读取再加载最新数据，逻辑更简单。

### 2. Read Through

应用只读缓存，缓存组件负责从数据库加载。

### 3. Write Through

写缓存时同步写数据库。

### 4. Write Behind

先写缓存，异步写数据库。

吞吐高，但一致性和数据丢失风险更大，核心交易数据慎用。

## 缓存一致性

缓存和数据库很难做到绝对强一致，通常追求最终一致。

常见做法：

- 更新数据库后删除缓存。
- 设置合理 TTL。
- 使用消息通知删除缓存。
- 使用延迟双删。
- 对核心数据直接读数据库。

订单和库存这类强一致数据，不应该完全依赖缓存判断最终结果。缓存可以加速查询，但写入和扣减要落数据库或专门库存系统。

## 缓存穿透

查询一个不存在的数据，每次都打到数据库。

解决：

- 缓存空值。
- 布隆过滤器。
- 参数校验。
- 限流。

示例：

```text
tracking_no 不存在
每次查询缓存都 miss
每次都查数据库
```

可以缓存空结果短 TTL。

## 缓存击穿

某个热点 key 过期，大量请求同时打到数据库。

解决：

- 互斥锁重建缓存。
- singleflight 合并请求。
- 热点 key 永不过期加后台刷新。
- 提前异步刷新。

## 缓存雪崩

大量 key 同时过期，导致数据库瞬间压力暴增。

解决：

- TTL 加随机抖动。
- 分批预热。
- 多级缓存。
- 限流降级。

示例：

```go
ttl := baseTTL + time.Duration(rand.Intn(300))*time.Second
```

## 热点 key

某个 key 被大量访问，例如爆款商品库存、热门物流轨迹。

解决：

- 本地缓存。
- 多副本缓存。
- key 拆分。
- 请求合并。
- 限流。

## 大 key

value 太大或集合元素太多，会影响 Redis 性能。

风险：

- 网络传输慢。
- 阻塞 Redis。
- 删除成本高。
- 内存碎片。

解决：

- 拆 key。
- 分页读取。
- 控制 value 大小。
- 避免一个 hash/list/zset 无限增长。

## 面试回答模板

缓存主要用于提升读性能和保护数据库。跨境物流里，我会缓存轨迹最新状态、承运商配置、国家地区配置、运费模板等读多写少的数据。Redis 比 Memcached 功能更丰富，实际项目中更常用。

缓存模式上常用 Cache Aside：读时先查缓存，miss 后查库并写缓存；写时先更新数据库，再删除缓存。因为缓存和数据库很难强一致，所以要配合 TTL、消息删除、重试和必要的对账。

缓存还要处理穿透、击穿、雪崩。穿透可以缓存空值或用布隆过滤器；击穿可以用互斥锁或 singleflight；雪崩可以给 TTL 加随机抖动。对于订单库存这种强一致数据，缓存只能辅助查询，不能作为最终扣减依据。

## 常见追问

### 1. 更新数据时为什么删除缓存而不是更新缓存？

删除缓存更简单，能减少并发覆盖问题。更新缓存时，如果两个写操作乱序，可能旧值覆盖新值。

### 2. Redis 能不能做分布式锁？

可以，但要谨慎。简单场景可以用 `SET key value NX EX seconds`，释放时要校验 value。强一致锁场景要评估 Redis 故障和主从切换风险。

### 3. 库存能不能直接放 Redis 扣？

秒杀类库存可以用 Redis 做前置扣减和限流，但最终仍要和数据库或库存系统对账，保证最终一致。核心是原子扣减、幂等、补偿和防超卖。
