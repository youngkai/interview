# 排序、搜索、查找优化

## 30 秒回答

排序和查找优化的核心是根据数据规模、数据特征和访问模式选择合适算法和数据结构。小数据量直接排序或线性查找可能足够；大量数据要考虑索引、哈希表、二分查找、堆、倒排索引、数据库索引、缓存和分片。

面试中不要只背快排、归并、堆排复杂度，要能说清楚工程场景：订单按时间分页用索引，查订单详情用哈希或主键索引，Top K 用堆，范围查询用 B+ 树，全文搜索用倒排索引。

## 排序算法对比

| 算法 | 平均复杂度 | 最坏复杂度 | 稳定性 | 适合场景 |
| --- | --- | --- | --- | --- |
| 快速排序 | O(n log n) | O(n^2) | 不稳定 | 通用内存排序 |
| 归并排序 | O(n log n) | O(n log n) | 稳定 | 外部排序、链表排序 |
| 堆排序 | O(n log n) | O(n log n) | 不稳定 | Top K、优先队列 |
| 插入排序 | O(n^2) | O(n^2) | 稳定 | 小数组、近乎有序 |
| 计数排序 | O(n+k) | O(n+k) | 稳定 | 整数范围较小 |

Go 标准库的 `sort` 对普通业务已经足够，不建议自己手写排序，除非面试题要求。

## 工程中的排序场景

### 1. 订单列表按时间排序

数据库中常见：

```sql
SELECT *
FROM orders
WHERE user_id = ?
ORDER BY created_at DESC
LIMIT 20;
```

要建立合适索引：

```text
(user_id, created_at)
```

否则数据量大时会产生慢查询。

### 2. Top K

例如找出访问量最高的 100 个商品。

如果数据量很大，不需要全量排序，可以用小顶堆维护 K 个最大值。

复杂度：

```text
O(n log k)
```

比全量排序：

```text
O(n log n)
```

更合适。

### 3. 多路归并

多个分片各自有序，需要合并成全局有序。

例如分库分表后查询多个订单分片，每个分片返回最近 20 条，再做多路归并。

## 查找方式对比

| 查找方式 | 复杂度 | 前提 |
| --- | --- | --- |
| 线性查找 | O(n) | 无要求 |
| 二分查找 | O(log n) | 数据有序 |
| 哈希查找 | 平均 O(1) | 有 hash 结构 |
| B+ 树索引 | O(log n) | 数据库索引 |
| 倒排索引 | 依赖实现 | 文本搜索 |

## 查找优化场景

### 1. 用 map 替代循环查找

低效：

```go
for _, order := range orders {
    if order.ID == targetID {
        return order
    }
}
```

如果频繁按 ID 查找，应该建立 map：

```go
orderByID := make(map[int64]Order, len(orders))
for _, order := range orders {
    orderByID[order.ID] = order
}
```

### 2. 二分查找

数据有序时使用二分。

```go
idx := sort.Search(len(nums), func(i int) bool {
    return nums[i] >= target
})
```

适合：

- 有序数组。
- 时间线查找。
- 配置区间匹配。
- 版本号查找。

### 3. 数据库索引

数据库查找优化的重点不是在应用层循环，而是让数据库走索引。

常见索引：

- 主键索引。
- 唯一索引。
- 联合索引。
- 覆盖索引。
- 前缀索引。

### 4. 缓存

热点数据可以缓存：

- 订单详情。
- 物流最新状态。
- 承运商配置。
- 国家地区配置。

但缓存要处理一致性和过期。

### 5. 搜索引擎

复杂搜索不要硬压数据库。

例如：

- 按收件人姓名模糊搜索。
- 多条件筛选订单。
- 全文搜索地址。
- 聚合统计。

可以用 Elasticsearch/OpenSearch 等。

## 分页优化

深分页低效：

```sql
SELECT * FROM orders ORDER BY created_at DESC LIMIT 100000, 20;
```

数据库需要跳过大量数据。

优化方式：

### 游标分页

```sql
SELECT *
FROM orders
WHERE user_id = ?
  AND created_at < ?
ORDER BY created_at DESC
LIMIT 20;
```

使用上一次最后一条记录的 `created_at` 或 `order_id` 作为游标。

## 面试回答模板

排序和查找优化要先看场景。数据量小的时候，直接使用标准库排序和线性查找可能就够了。数据量大或查询频繁时，要换数据结构，比如用 map 把 O(n) 查找变成平均 O(1)，有序数据用二分，Top K 用堆，数据库查询靠索引，全文检索交给搜索引擎。

在订单系统里，订单详情通常走 order_id 主键索引；用户订单列表要建立 `(user_id, created_at)` 联合索引；深分页用游标分页；分库分表后的全局排序可以让每个分片先返回局部有序结果，再做多路归并。

## 常见追问

### 1. Top K 为什么用堆？

因为只需要 K 个最大或最小值，不需要全量排序。维护一个大小为 K 的堆，复杂度是 O(n log k)。

### 2. 二分查找前提是什么？

数据必须有序，并且能随机访问。链表不适合二分，因为定位中点不是 O(1)。

### 3. 数据库索引越多越好吗？

不是。索引能提升查询，但会增加写入成本、存储空间和维护成本。要根据查询模式设计。
