# GMP 调度模型

## 30 秒回答

GMP 是 Go runtime 的调度模型：G 是 Goroutine，M 是操作系统线程，P 是 Processor，也就是执行 Go 代码所需的调度上下文。Goroutine 要运行，必须由 M 绑定 P 后执行。P 的数量默认等于 `GOMAXPROCS`，决定了同一时刻最多有多少个线程并行执行 Go 代码。

GMP 的核心目标是用少量 OS 线程高效调度大量 Goroutine。它通过本地队列、全局队列、work stealing、netpoller、抢占调度等机制提升并发性能和资源利用率。

## G、M、P 分别是什么

### G：Goroutine

G 是 Go 代码中的并发任务。每个 G 保存自己的栈、状态、待执行函数、调度信息等。

常见状态包括：

- `_Grunnable`：可运行，等待调度。
- `_Grunning`：正在执行。
- `_Gwaiting`：等待中，例如 channel、锁、网络 I/O、定时器。
- `_Gsyscall`：系统调用中。
- `_Gdead`：已结束。

### M：Machine

M 是 OS 线程。真正执行代码的是线程，Goroutine 必须跑在线程上。

M 可能处于：

- 正在执行 Go 代码。
- 正在系统调用。
- 空闲等待。
- 被 runtime 创建或回收。

### P：Processor

P 是 Go 调度器的处理器上下文。M 想执行 Go 代码，必须先持有一个 P。

P 里有：

- 本地 runnable G 队列。
- 内存分配缓存。
- 调度相关状态。

`GOMAXPROCS` 控制 P 的数量，也就控制了同时执行 Go 代码的最大并行度。

## 为什么需要 P

早期如果只有 G 和 M，多个线程竞争全局队列，锁竞争会比较重。P 的引入让每个 P 维护本地队列，大部分调度在本地完成，降低全局锁竞争。

P 还承载了运行 Go 代码所需的一些资源，例如内存分配缓存。这样 M 绑定 P 后就可以高效运行。

## 调度流程

一个典型流程：

1. 程序通过 `go func()` 创建一个 G。
2. 新 G 优先放入当前 P 的本地队列。
3. M 绑定 P，从 P 的本地队列取 G 执行。
4. 如果本地队列没有 G，M 会尝试从全局队列取。
5. 如果全局队列也没有，M 会从其他 P 的本地队列偷一部分 G。
6. 如果还是没有，会检查 netpoller 中是否有就绪的网络 I/O。
7. 没有任务时，M 进入休眠或自旋等待。

## 本地队列和全局队列

每个 P 都有自己的本地队列，减少多个线程争抢同一把锁的情况。

全局队列用于存放一些无法放入本地队列的 G，或者某些调度路径产生的 G。

调度器会定期从全局队列取 G，避免全局队列里的任务长期饥饿。

## Work Stealing

当某个 P 的本地队列为空时，它不会立刻休眠，而是尝试从其他 P 的本地队列偷一部分 G 来执行。这叫 work stealing。

好处：

- 让各个 P 的负载更均衡。
- 避免某些 P 很忙、某些 P 空闲。
- 提升 CPU 利用率。

## 系统调用时发生什么

如果 G 进入系统调用，对应 M 可能会阻塞。此时：

1. G 进入 `_Gsyscall`。
2. M 和 P 解绑。
3. P 可以交给其他 M 继续执行其他 G。
4. 系统调用返回后，原来的 G 需要重新获得 P 才能继续执行。

这样可以避免一个阻塞系统调用占住 P，影响其他 Goroutine 执行。

## 网络 I/O 和 netpoller

Go 的网络库和 runtime netpoller 配合。Goroutine 等待网络 I/O 时，runtime 会把它挂起，并把 fd 注册到 netpoller。

当 I/O 就绪后，netpoller 会把对应 G 重新放回可运行队列。

这使得 Go 能用同步代码写法处理大量网络连接，而不是每个连接都占一个 OS 线程阻塞等待。

## 抢占调度

早期 Go 的调度更多依赖协作式让出，例如函数调用、channel 操作、系统调用等。后来 Go 引入了更完善的异步抢占能力，避免某个 Goroutine 长时间占用 P。

抢占调度解决的问题：

- 死循环或长时间计算导致其他 Goroutine 饿死。
- GC 需要等待 Goroutine 到达安全点。
- 提升整体调度公平性。

示例：

```go
for {
    // 如果没有抢占机制，这类循环可能长期占住执行资源。
}
```

现代 Go 中 runtime 能在更多位置打断长时间运行的 Goroutine。

## GOMAXPROCS 的意义

`GOMAXPROCS` 决定 P 的数量，也就是同一时刻最多有多少个 M 可以并行执行 Go 代码。

```go
runtime.GOMAXPROCS(4)
```

注意：

- 它不是 Goroutine 数量上限。
- 它不是线程数量上限。
- 它控制的是 Go 代码并行执行的最大宽度。

CPU 密集型任务通常受 `GOMAXPROCS` 影响比较明显；I/O 密集型任务更多受网络、数据库、下游服务、连接池和调度影响。

## 面试回答模板

GMP 模型里，G 是 Goroutine，M 是 OS 线程，P 是执行 Go 代码需要的上下文。M 必须绑定 P 才能运行 G。P 的数量由 `GOMAXPROCS` 决定，代表 Go 代码的最大并行度。

当创建 Goroutine 时，G 通常会进入当前 P 的本地队列。M 从本地队列取 G 执行；如果没有任务，会尝试全局队列、其他 P 的队列，也就是 work stealing；网络 I/O 就绪的任务会通过 netpoller 回到可运行队列。

如果 Goroutine 进入系统调用，M 可能阻塞，但 P 会释放给其他 M，避免影响其他 Goroutine 执行。这个模型的价值是减少内核线程数量、降低调度成本、提升高并发场景下的吞吐。

## 常见追问

### 1. P 的数量越多越好吗？

不是。P 决定并行执行 Go 代码的数量。CPU 密集型任务中，P 太多会增加上下文切换和调度开销；I/O 密集型任务中，瓶颈可能不在 P，而在下游、连接池、锁或 GC。

### 2. Go 线程数量会超过 GOMAXPROCS 吗？

会。`GOMAXPROCS` 控制 P 的数量，不是 OS 线程数量。系统调用、cgo、runtime 需求都可能让 M 数量超过 P 数量。

### 3. 为什么 work stealing 能提升性能？

因为不同 P 的本地队列任务量可能不均衡。空闲 P 从忙碌 P 偷任务，可以提升 CPU 利用率，减少任务等待时间。

### 4. Goroutine 是公平调度吗？

Go 调度器会尽量保证公平，例如全局队列检查、抢占调度、netpoller 唤醒等，但不能简单理解为严格公平。实际表现还会受到锁竞争、channel 设计、系统调用和 CPU 密集计算影响。
