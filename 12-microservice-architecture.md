# 微服务架构

## 30 秒回答

微服务架构是把一个复杂系统按业务边界拆成多个可独立开发、部署、扩缩容的服务。跨境物流里常见拆分包括订单服务、库存服务、仓储服务、运输服务、轨迹服务、计费服务、支付服务、通知服务等。

微服务的价值是降低单体系统复杂度，让不同业务模块可以独立演进和扩容。但它也会带来分布式调用、数据一致性、链路追踪、服务治理、部署复杂度等问题。所以微服务不是越拆越好，核心是按业务边界和团队协作边界做合理拆分。

## 跨境物流中的典型服务拆分

| 服务 | 职责 |
| --- | --- |
| 订单服务 | 创建订单、取消订单、订单主状态流转 |
| 库存服务 | 库存查询、库存预占、库存扣减、库存释放 |
| 仓储服务 | 入库、出库、拣货、打包、交接 |
| 运输服务 | 运输计划、承运商分配、运单创建 |
| 轨迹服务 | 物流节点、清关状态、签收状态同步 |
| 支付服务 | 支付、退款、对账 |
| 计费服务 | 运费计算、附加费、税费估算 |
| 通知服务 | 邮件、短信、站内信、Webhook |
| 用户服务 | 用户资料、地址、权限 |

## 微服务拆分原则

### 1. 按业务能力拆分

不要按技术层次拆分，例如 user-controller、user-service、user-dao 这种不是微服务拆分。

更合理的是按业务能力：

```text
订单服务
库存服务
运输服务
轨迹服务
结算服务
```

每个服务都拥有自己的业务逻辑和数据。

### 2. 高内聚、低耦合

服务内部职责要完整，服务之间依赖要清晰。

如果一个业务动作每次都要跨 5 个服务同步调用，可能说明拆得太碎，或者服务边界不合理。

### 3. 数据归属清晰

一个数据对象应该有明确 owner。

例如：

- 订单主表归订单服务。
- 库存数量归库存服务。
- 物流轨迹归轨迹服务。

其他服务不要直接跨库改数据，而是通过接口或事件协作。

### 4. 避免一开始过度拆分

如果业务还不稳定，过早拆成很多服务会增加沟通、部署、排障成本。

更实际的做法：

1. 先按模块写清楚边界。
2. 数据表和代码结构上预留拆分空间。
3. 等业务规模、团队规模、性能瓶颈明确后再拆服务。

## 微服务常见通信方式

### 同步调用

常见于需要立即返回结果的场景：

- 下单时查询库存。
- 创建运单时调用承运商报价。
- 支付时调用支付网关。

可选技术：

- HTTP REST。
- gRPC。
- Thrift。

### 异步事件

常见于不要求调用方立刻拿到最终结果的场景：

- 订单创建后发送通知。
- 库存扣减后异步更新搜索索引。
- 物流轨迹更新后通知用户。
- 支付成功后触发履约流程。

可选技术：

- Kafka。
- RabbitMQ。
- RocketMQ。
- SQS 等云消息队列。

## 微服务的基础治理能力

### 1. 服务注册与发现

调用方不应该写死 IP，需要通过注册中心或服务网格找到服务实例。

### 2. 配置管理

配置需要集中管理，并支持环境隔离、动态更新和审计。

### 3. 链路追踪

一个跨境订单可能经过订单、库存、仓储、运输、轨迹、通知多个服务。必须有 trace id 贯穿全链路，方便排查问题。

### 4. 熔断、限流、降级

下游异常时，不能让故障扩散到整个系统。

例如承运商报价服务变慢时，订单服务可以：

- 返回默认报价。
- 提示稍后重试。
- 使用缓存报价。
- 限制调用并发。

### 5. 监控和告警

核心指标：

- QPS。
- 错误率。
- P95/P99 延迟。
- CPU、内存、GC。
- 队列积压。
- 数据库连接池。
- 下游调用耗时。

## 微服务的代价

微服务不是免费收益，常见代价包括：

- 网络调用失败。
- 分布式事务复杂。
- 数据一致性难。
- 本地调试复杂。
- 部署和发布链路变长。
- 链路排查难。
- 服务版本兼容问题。
- 测试环境依赖多。

所以面试回答时要体现权衡：不是为了微服务而微服务，而是为了解决业务复杂度、团队协作和独立扩展问题。

## 面试回答模板

跨境物流系统可以按业务能力拆成订单、库存、仓储、运输、轨迹、计费、支付、通知等服务。每个服务负责自己的核心数据和业务规则，比如库存数量只能由库存服务维护，订单主状态由订单服务维护。

服务之间同步调用用于需要立即结果的场景，比如下单时校验库存；异步消息用于最终一致和解耦场景，比如订单创建后通知仓储、更新轨迹、发送通知。

微服务带来的问题是分布式调用、数据一致性、链路追踪和服务治理。所以设计时要配套注册发现、限流熔断、超时重试、幂等、监控告警和消息补偿机制。服务也不能拆得太碎，要按业务边界和团队边界拆。

## 常见追问

### 1. 微服务拆得越细越好吗？

不是。拆太细会导致调用链过长、事务复杂、部署和排障成本增加。要按业务边界、数据归属、团队协作和扩展需求来拆。

### 2. 服务之间能不能直接访问对方数据库？

一般不建议。跨服务直接访问数据库会破坏服务边界，导致数据 owner 不清晰，也会增加变更风险。应该通过接口或事件协作。

### 3. 微服务如何排查问题？

依赖 trace id、结构化日志、指标监控、链路追踪和告警。每次请求都要能看到经过哪些服务、每个服务耗时多少、错误发生在哪里。
