# 高并发和高可用

## 30 秒回答

高并发关注系统能不能扛住大量请求，高可用关注系统在部分组件失败时能不能继续提供服务。跨境物流系统中，下单、查轨迹、库存扣减、承运商回调都可能出现高并发。

设计时要从入口限流、水平扩展、缓存、异步削峰、数据库优化、服务降级、熔断、超时重试、隔离、监控告警和容灾多活等方面考虑。核心原则是给系统加边界，避免流量把某个点打穿后级联故障。

## 高并发设计思路

### 1. 水平扩展

服务尽量无状态化，多个实例部署在负载均衡后面。

无状态服务更容易扩容：

```text
LB -> order-service-1
   -> order-service-2
   -> order-service-3
```

状态放到数据库、缓存、消息队列或对象存储中。

### 2. 限流

限制入口流量，防止系统被打爆。

常见限流维度：

- 全局 QPS。
- 单用户 QPS。
- 单商家 QPS。
- 单 IP QPS。
- 单接口 QPS。

跨境物流中，开放平台 API 必须做租户级限流，避免某个商家流量影响其他商家。

### 3. 削峰填谷

使用消息队列承接突发流量。

例如订单创建后，后续通知仓储、生成运单、发送邮件不一定同步完成，可以投递 MQ 异步处理。

### 4. 缓存

热点读请求可以走缓存。

适合缓存：

- 商品信息。
- 物流轨迹最近状态。
- 承运商配置。
- 国家地区配置。
- 运费模板。

不适合随意缓存：

- 强一致库存。
- 支付金额最终结果。
- 频繁变更且一致性要求高的数据。

### 5. 数据库优化

常见手段：

- 索引优化。
- 读写分离。
- 分库分表。
- 批量写入。
- 避免大事务。
- 避免慢查询。
- 控制连接池。

### 6. 异步化

同步链路越长，请求越容易超时。

下单主链路只保留必要步骤：

```text
校验参数 -> 校验库存 -> 创建订单 -> 预占库存 -> 返回
```

非核心步骤异步：

```text
发送通知
更新搜索索引
生成报表
同步 BI
```

## 高可用设计思路

### 1. 超时

所有外部调用都要设置超时。没有超时的调用可能永久占用资源。

### 2. 重试

重试只适合短暂故障，并且必须配合：

- 超时。
- 最大次数。
- 退避策略。
- 幂等。

不能无限重试，否则会放大故障。

### 3. 熔断

下游持续失败时，暂时停止调用，快速失败或降级，保护系统。

### 4. 降级

非核心能力失败时，不影响主流程。

例如：

- 物流轨迹详情暂时不可用时，只展示最新状态。
- 承运商报价失败时，展示缓存报价或提示稍后重试。
- 通知服务失败时，写入重试队列。

### 5. 隔离

避免不同业务互相拖垮。

隔离方式：

- 线程池或 worker pool 隔离。
- 数据库连接池隔离。
- Redis key 空间隔离。
- 租户限流隔离。
- 核心和非核心服务分开部署。

### 6. 多副本和故障转移

核心组件需要多副本：

- 服务实例多副本。
- 数据库主从或集群。
- Redis 哨兵或集群。
- MQ 集群。

### 7. 监控告警

高可用不是只靠设计，必须靠监控发现问题。

核心指标：

- QPS。
- 错误率。
- P95/P99 延迟。
- CPU、内存、GC。
- Goroutine 数量。
- DB 慢查询。
- 连接池使用率。
- MQ 积压。
- Redis 命中率。

## 面试回答模板

高并发和高可用我会分层设计。入口层用负载均衡和限流，服务层无状态化方便水平扩展，核心链路尽量短，非核心动作通过 MQ 异步削峰。读多的数据用缓存，数据库做索引优化、读写分离或分库分表。

高可用方面，所有下游调用必须有超时、重试和熔断，重试要有次数限制和幂等保障。非核心能力要能降级，核心资源要做隔离，避免一个慢下游拖垮整个服务。最后通过监控告警、链路追踪和压测验证系统瓶颈。

跨境物流场景里，我会特别关注订单、库存、轨迹这些核心链路。比如轨迹更新可以异步最终一致，但库存扣减要严格控制并发和幂等，避免超卖。

## 常见追问

### 1. 高并发一定要分库分表吗？

不一定。先看瓶颈在哪里。很多时候缓存、索引、读写分离、异步化和限流就能解决。分库分表会增加复杂度，应该在单库确实成为瓶颈时再做。

### 2. 重试为什么可能有风险？

如果接口不幂等，重试可能导致重复扣库存、重复支付、重复创建运单。即使幂等，过度重试也会放大下游压力。

### 3. 如何防止级联故障？

用超时、熔断、限流、降级和资源隔离。下游慢或失败时，上游不能无限等待和重试。
