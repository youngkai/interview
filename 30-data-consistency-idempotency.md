# 数据一致性和幂等性处理

## 30 秒回答

数据一致性关注多个系统、多个表、缓存和消息之间的数据最终是否正确；幂等性关注同一个请求或消息重复执行时，结果是否只生效一次。跨境物流中订单创建、库存扣减、支付回调、物流状态回调、退款处理都必须考虑幂等。

常见手段包括唯一约束、幂等表、request_id/event_id、状态机、乐观锁、条件更新、本地消息表、事务消息、重试补偿和定时对账。面试回答时要强调：分布式系统里重复、乱序、超时、部分失败都是常态，不能假设请求只来一次。

## 为什么需要幂等

分布式系统中这些情况很常见：

- 调用方超时后重试。
- MQ 重复投递。
- 支付平台重复回调。
- 承运商重复推送物流轨迹。
- 用户重复点击提交。
- 服务处理成功但响应丢失。

如果没有幂等，可能发生：

- 重复创建订单。
- 重复扣库存。
- 重复退款。
- 订单状态回退。
- 重复发送通知。

## 幂等设计常用方案

### 1. 数据库唯一约束

防重复创建订单：

```text
unique(tenant_id, external_order_no)
```

防用户重复秒杀：

```text
unique(activity_id, sku_id, user_id)
```

防重复处理消息：

```text
unique(event_id)
```

唯一约束是很可靠的最后防线。

### 2. 幂等表

保存请求处理记录：

```text
idempotency_key
status
result
created_at
updated_at
```

流程：

1. 请求进来先插入幂等 key。
2. 插入成功，执行业务。
3. 业务成功后记录结果。
4. 重复请求直接返回已有结果。

适合：

- 支付。
- 创建订单。
- 创建运单。
- 退款。

### 3. 状态机

订单状态只能按合法路径流转。

```text
WAIT_PAY -> PAID -> SHIPPED -> DELIVERED
WAIT_PAY -> CANCELED
```

重复或非法状态直接忽略或拒绝。

例如签收事件重复到达：

```text
DELIVERED -> DELIVERED
```

可以视为幂等成功。

### 4. 条件更新

支付成功：

```sql
UPDATE orders
SET status = 'PAID'
WHERE order_id = ?
  AND status = 'WAIT_PAY';
```

只有待支付订单能更新为已支付。重复支付回调影响行数为 0，再查询当前状态判断是否已经成功。

取消订单：

```sql
UPDATE orders
SET status = 'CANCELED'
WHERE order_id = ?
  AND status = 'WAIT_PAY';
```

支付和取消并发时，谁先成功，另一个会失败，避免状态互相覆盖。

### 5. 乐观锁

```sql
UPDATE orders
SET status = ?, version = version + 1
WHERE order_id = ?
  AND version = ?;
```

防止并发覆盖。

### 6. 消息去重

消费者处理消息前记录 event_id。

```sql
INSERT INTO processed_event(event_id, created_at)
VALUES (?, ?);
```

唯一键冲突说明处理过，直接 ack。

## 数据一致性处理

### 1. 本地事务

单服务内尽量用本地事务保证原子性。

```text
begin
  创建订单
  写订单状态流水
commit
```

### 2. 本地消息表

解决“业务数据写成功，但消息发送失败”的问题。

```text
begin
  更新订单
  插入 outbox_message
commit
```

后台任务扫描消息表发送 MQ。

### 3. 事务消息

如果 MQ 支持事务消息，可以保证本地事务和消息发送的一致性关系。

### 4. Saga 补偿

长流程拆成多个本地事务，失败时执行补偿。

```text
创建订单 -> 预占库存 -> 创建运单
失败补偿：取消运单 -> 释放库存 -> 取消订单
```

### 5. TCC

适合资源预留类场景：

```text
Try: 预占库存
Confirm: 确认扣减
Cancel: 释放库存
```

### 6. 对账

定时对账是最终一致的兜底。

跨境物流对账例子：

- 支付平台已支付，订单仍待支付。
- 承运商已签收，订单仍运输中。
- 库存预占成功，订单创建失败。
- 退款成功，订单售后状态未更新。

## 缓存一致性

常见做法：

```text
更新数据库 -> 删除缓存
```

删除失败要重试，并设置 TTL 兜底。

如果对一致性要求很高，可以：

- 直接读数据库。
- 使用较短 TTL。
- 通过 MQ/CDC 删除缓存。
- 对关键状态不缓存或只缓存只读副本。

## 乱序处理

物流事件和 MQ 消息可能乱序。

处理方式：

- event_time。
- version。
- sequence。
- 状态机。
- 主状态只向前推进。
- 明细事件完整保存。

例子：

```text
Delivered 先到
OutForDelivery 后到
```

订单主状态不能从 DELIVERED 回退到 OUT_FOR_DELIVERY。

## 面试回答模板

数据一致性和幂等性在跨境物流里非常关键，因为订单、库存、支付、运输、清关都是分布式协作，请求可能超时、消息可能重复、事件可能乱序。我的思路是先保证单服务内本地事务正确，再通过消息和补偿保证跨服务最终一致。

幂等方面，创建类接口用 idempotency key 或业务唯一键，数据库唯一约束兜底；状态更新用状态机和条件更新；消息消费用 event_id 去重表；支付和物流回调都按重复回调处理。库存扣减用条件更新或 TCC 预占，避免超卖。

一致性方面，业务数据和消息发送用本地消息表或事务消息，消费者幂等处理，失败重试，多次失败进入死信队列，最后用定时对账发现和修复异常数据。

## 常见追问

### 1. 幂等键应该怎么生成？

可以由调用方生成 request_id，也可以用业务唯一键组合，例如 `tenant_id + external_order_no`、`payment_no`、`event_id`。关键是同一次业务操作重复请求时 key 必须相同。

### 2. 重复请求直接返回成功吗？

要看第一次处理结果。如果第一次成功，可以返回成功结果；如果处理中，可以返回处理中；如果失败，要根据失败类型决定是否允许重试。

### 3. 状态机为什么能保证幂等？

状态机限制合法流转，重复事件不会重复产生副作用，过期或乱序事件不能让状态回退。
