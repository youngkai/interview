# 消息队列：Kafka 和 RabbitMQ 用在什么场景

## 30 秒回答

消息队列主要用于异步解耦、削峰填谷、最终一致性、事件通知和日志流处理。跨境物流中，订单创建后通知仓储、库存变更通知、物流轨迹同步、邮件短信通知、报表统计都适合用 MQ。

Kafka 更适合高吞吐、日志流、事件流、可回放场景；RabbitMQ 更适合传统业务消息、灵活路由、低延迟任务分发。无论使用哪种 MQ，都要处理消息重复、乱序、积压、失败重试、死信队列和消费者幂等。

## MQ 的核心价值

### 1. 异步解耦

订单服务不需要直接调用所有下游。

```text
订单服务 -> OrderCreated 事件 -> MQ
仓储服务消费
通知服务消费
BI 服务消费
```

新增消费者时，订单服务无需修改。

### 2. 削峰填谷

高峰期请求先进入 MQ，消费者按能力处理，避免瞬时流量打爆下游。

### 3. 最终一致性

跨服务数据同步可以通过事件实现最终一致。

### 4. 失败重试

消费者失败后可以重试、延迟重试或进入死信队列。

### 5. 流式处理

大量日志、轨迹事件、操作事件可以进入 Kafka，供多个系统消费。

## Kafka 适合场景

Kafka 的特点：

- 高吞吐。
- 持久化日志。
- 分区有序。
- 消费者按 offset 消费。
- 消息可回放。
- 适合事件流。

适合：

- 物流轨迹事件流。
- 订单状态变更事件。
- 用户行为日志。
- BI 数据同步。
- 搜索索引更新。
- CDC 数据同步。

示例：

```text
topic: logistics_tracking_events
key: tracking_no
value: event payload
```

用 `tracking_no` 作为 key，可以保证同一个包裹的事件进入同一个 partition，从而在 partition 内有序。

## RabbitMQ 适合场景

RabbitMQ 的特点：

- 路由能力强。
- exchange、queue、routing key 灵活。
- 支持 ack、死信、延迟插件等。
- 适合业务任务分发。

适合：

- 订单通知。
- 邮件短信任务。
- 延迟取消订单。
- 重试队列。
- 任务分发。

示例：

```text
exchange: order.events
routing_key: order.created
queue: warehouse.create_task
```

## MQ 常见问题

### 1. 消息重复

MQ 通常更容易做到至少一次投递，而不是严格只投一次。

消费者必须幂等。

幂等方式：

- event_id 去重。
- 业务唯一键。
- 数据库唯一索引。
- 状态机判断。

### 2. 消息丢失

要从三段看：

```text
生产者 -> MQ -> 消费者
```

生产者：

- 发送确认。
- 本地消息表。
- 事务消息。

MQ：

- 持久化。
- 多副本。
- 合理 ack 配置。

消费者：

- 处理成功后再 ack。
- 失败不 ack 或进入重试。

### 3. 消息乱序

Kafka 只能保证单 partition 内有序。RabbitMQ 多消费者也可能导致处理顺序变化。

解决：

- 同一业务 key 进入同一 partition。
- 单 key 串行消费。
- 状态机防乱序。
- 事件版本号。

### 4. 消息积压

原因：

- 消费者处理慢。
- 下游故障。
- 消费者实例太少。
- 单条消息处理异常反复重试。

解决：

- 扩容消费者。
- 批量消费。
- 优化处理逻辑。
- 跳过或隔离毒性消息。
- 临时降级非核心消费。

### 5. 死信队列

多次消费失败的消息进入死信队列，等待人工排查或补偿任务处理。

## 面试回答模板

消息队列在跨境物流系统里主要用于异步解耦、削峰填谷和最终一致性。比如订单创建后，订单服务只需要写本地订单并发送 OrderCreated 事件，仓储、通知、BI、搜索索引服务各自消费，不需要订单服务同步调用所有下游。

Kafka 更适合高吞吐事件流，比如物流轨迹、订单状态变更、日志和 CDC；RabbitMQ 更适合业务任务分发、灵活路由、延迟任务和通知类场景。

使用 MQ 的关键不是只会发送消息，而是要处理重复、丢失、乱序和积压。消费者必须幂等；生产者要用本地消息表或事务消息保证业务数据和消息一致；失败消息要有重试和死信队列；状态更新要用状态机防止乱序事件改错状态。

## 常见追问

### 1. MQ 如何保证消息不丢？

生产端要确认发送成功，必要时用本地消息表或事务消息；MQ 端要持久化和多副本；消费端要处理成功后再 ack。

### 2. 消费者为什么要幂等？

因为消息可能重复投递，消费者处理成功但 ack 失败时，MQ 可能再次投递同一消息。

### 3. Kafka 和 RabbitMQ 怎么选？

高吞吐事件流、日志、可回放选 Kafka；业务任务、复杂路由、传统队列模型可以选 RabbitMQ。最终还要看团队运维能力和现有技术栈。
